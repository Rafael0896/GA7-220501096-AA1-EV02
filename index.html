Estándares de Codificación

Lenguajes utilizados en el proyecto

  • HTML (HyperText Markup Language): Define la estructura y contenido de la página web utilizando HTML5 con etiquetas semánticas.
  • CSS (Cascading Style Sheets): Define la presentación visual incluyendo colores, tipografía, espaciado, y diseño responsive.
  • JavaScript: Añade interactividad a la página, maneja eventos y mejora la experiencia de usuario.

Estándares HTML

  • Declaración DOCTYPE: Siempre incluir para asegurar la interpretación correcta del HTML5.
  • Etiqueta lang: Especificar el idioma del documento para accesibilidad y SEO.
  • Meta tags: Incluir meta etiquetas esenciales para codificación y visualización.
  • Estructura semántica: Usar etiquetas semánticas en lugar de divs genéricos.
  • Indentación: 4 espacios para cada nivel de anidamiento.
<!DOCTYPE html>
<html lang="es">
    <head>
        <meta charset="UTF-8">
        <meta name="viewport" content="width=device-width, initial-scale=1.0">
    </head>

Estándares CSS

  • Variables CSS: Utilizar para valores reusables como colores y tamaños.
  • Organización: Agrupar estilos relacionados y comentar secciones principales.
  • Media Queries: Implementar para asegurar diseño responsive.
  • Selectores: Mantener selectores simples y evitar excesiva especificidad.
  • Reset/Normalize: Incluir reset CSS para consistencia entre navegadores.
:root {
    --color-primary: #3498db;
    --color-secondary: #2ecc71;
}

/* Estilos para navegación */
.nav-item {
    margin-left: 30px;
}

Estándares JavaScript

  • "use strict": Usar modo estricto para evitar errores comunes.
  • Programación orientada a objetos: Utilizar clases para organizar el código.
  • Event listeners: Usar addEventListener en lugar de atributos HTML.
  • Contexto (this): Mantener el contexto correcto con .bind() o arrow functions.
  • DOMContentLoaded: Inicializar scripts cuando el DOM esté cargado.
"use strict";

class NavManager {
    constructor() {
        this.init();
    }

    init() {
        const navLinks = document.querySelectorAll('.nav-link');
        navLinks.forEach(link => {
            link.addEventListener('click', this.handleNavClick.bind(this));
        });
    }
}

Nomenclatura para variables y constantes

  • HTML - IDs: Usar camelCase o kebab-case consistentemente: mainContainer o main-container
  • HTML - Clases: Usar kebab-case: feature-card, nav-item
  • CSS - Variables: Usar kebab-case con prefijo --: --color-primary, --font-size-heading
  • JavaScript - Variables: Usar camelCase: let userName = 'Juan'
  • JavaScript - Constantes: Usar camelCase para instancias: const navManager = new NavManager()
  • JavaScript - Constantes globales: Usar UPPER_SNAKE_CASE: const API_KEY = 'abc123'

Nomenclatura para clases, métodos y comentarios

  • Clases: PascalCase (primera letra mayúscula): class NavManager {}
  • Métodos: camelCase, verbos o frases verbales: handleClick() {}
  • Comentarios HTML: <!-- Encabezado principal del sitio -->
  • Comentarios CSS: /* Estilos para la navegación principal */
  • Comentarios JavaScript:
/**
* Maneja el evento de clic en los enlaces de navegación
* @param {Event} event - El evento del navegador
* @return {void}
*/
handleNavClick(event) {
    // Implementación...
}

Buenas prácticas implementadas

  • Código limpio: Siguiendo principios de legibilidad y mantenibilidad.
  • Responsive Design: Media queries para asegurar buena experiencia en todos los dispositivos.
  • Accesibilidad: Etiquetas semánticas y contraste de color adecuado.

Control de Versiones con Git y GitHub

Configuración inicial del repositorio

# Inicializar un repositorio Git
git init

# Configurar información del usuario
git config --global user.name "Tu Nombre"
git config --global user.email "tu.email@ejemplo.com"

Flujo de trabajo básico

  1. Área de trabajo → Área de preparación → Repositorio local → Repositorio remoto
# Verificar el estado de los archivos
git status

# Añadir archivos al área de preparación
git add index.html
git add .

# Crear un commit con los cambios
git commit -m "Descripción clara del cambio realizado"

# Enviar cambios al repositorio remoto
git push origin main

Buenas prácticas para commits

  • Commits atómicos: Cada commit debe representar un único cambio lógico.
  • Mensajes descriptivos: Usar verbos en imperativo y explicar el "qué" y el "por qué".
  • Convención en mensajes:
    • feat: Nueva característica
    • fix: Corrección de error
    • docs: Cambios en documentación
    • style: Cambios de formato (no afectan el código)
    • refactor: Refactorización del código
    • test: Añadir o modificar pruebas
# Ejemplos de buenos mensajes de commit
git commit -m "feat: añadir sección de control de versiones"
git commit -m "fix: corregir enlaces de navegación en header"
git commit -m "docs: actualizar instrucciones de instalación"

Trabajando con ramas

# Crear y cambiar a una nueva rama
git checkout -b feature/nueva-seccion

# Listar todas las ramas
git branch

# Cambiar entre ramas
git checkout main

# Fusionar una rama con la rama actual
git merge feature/nueva-seccion

Estrategia de ramificación

  • main/master: Código estable y listo para producción
  • develop: Código en desarrollo para próxima versión
  • feature/nombre: Desarrollo de nuevas características
  • hotfix/nombre: Correcciones urgentes para producción
  • release/v1.x.x: Preparación de releases

Conexión con GitHub

# Agregar repositorio remoto
git remote add origin https://github.com/usuario/repositorio.git

# Verificar repositorios remotos
git remote -v

# Subir rama al repositorio remoto por primera vez
git push -u origin main

# Clonar un repositorio existente
git clone https://github.com/usuario/repositorio.git

Pull Requests

  1. Crear una rama: Desarrollar una característica en una rama separada
  2. Realizar commits: Hacer cambios y confirmarlos localmente
  3. Subir la rama: git push origin feature/nueva-seccion
  4. Crear PR en GitHub: Solicitar la fusión de la rama con main/master
  5. Revisión de código: Otros miembros revisan y comentan los cambios
  6. Realizar cambios: Si es necesario, hacer ajustes según los comentarios
  7. Fusionar: Una vez aprobado, el PR se fusiona en la rama principal

Resolución de conflictos

# Al intentar fusionar, pueden surgir conflictos
# Git marcará los archivos con conflictos

<<<<<<< HEAD
const VERSION = "1.1.0"; // Tu versión actual
=======
const VERSION = "1.2.0"; // Versión que intentas fusionar
>>>>>>> feature/nueva-version

# Editar el archivo para resolver el conflicto
# Añadir el archivo resuelto
git add archivo-con-conflicto.js
# Continuar la fusión
git commit

Comandos útiles adicionales

# Ver historial de commits
git log
git log --oneline --graph

# Deshacer cambios en el área de trabajo
git checkout -- index.html

# Deshacer el último commit (manteniendo cambios)
git reset --soft HEAD~1

# Deshacer cambios en el área de preparación
git reset HEAD index.html

# Crear y aplicar etiquetas (versiones)
git tag -a v1.0.0 -m "Primera versión estable"
git push origin v1.0.0

Recuerda

El control de versiones no solo sirve para guardar código, también es una herramienta de comunicación. Buenos commits y una estructura clara de ramas ayudan a otros desarrolladores (y a tu futuro yo) a entender la evolución del proyecto.
